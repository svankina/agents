# /arch_scan

Assess this project's architecture and record it in two places: the project's own `AGENTS.md` (concise, for future agents working here) and the canonical **architect** repo at `/home/svankina/src/architect` (the source of truth across projects).

`$ARGUMENTS` may contain a target path. If empty, use the current project.

Work through these steps in order. Run independent shell lookups together.

## 1. Resolve the project

- Project root: if `$ARGUMENTS` names a path, use it; else `git -C . rev-parse --show-toplevel` and fall back to the cwd if not a git repo.
- Slug: the basename of the project root, lowercased, non-alphanumerics → `-` (e.g. `My App` → `my-app`).
- Capture facts for the header: absolute path, git remote (`git -C <root> remote get-url origin` if any), current branch, and today's date.

## 2. Delegate the evaluation to the architect agent

Spawn the **architect** subagent (Agent tool, `subagent_type: "architect"`). Give it the absolute project root and slug, and ask it to return the concise architecture markdown per its output contract. Do not do the evaluation yourself — the subagent's returned markdown is the canonical content. Call its result `ARCH_MD` (it begins with a `## Architecture` heading).

## 3. Update the project's AGENTS.md

Target `<root>/AGENTS.md` (create it with a `# Agent Notes for <slug>` title if absent; if `AGENTS.md` is a symlink, edit the link target).

Maintain a managed block delimited by these exact markers so re-runs are idempotent:

```
<!-- arch_scan:start -->
<ARCH_MD here>

_Generated by `/arch_scan` on <date>. Source of truth: architect repo `projects/<slug>.md`._
<!-- arch_scan:end -->
```

- If the markers already exist, replace everything between them.
- If not, append the block (preceded by a blank line) to the end of the file.
- Leave all other content untouched.

## 4. Sync to the source of truth (architect repo)

Canonical store: `/home/svankina/src/architect/projects/<slug>.md`. Create the `projects/` directory if needed.

The file's body is `ARCH_MD`, prefixed with this header:

```markdown
# <slug>

> **Source:** `<absolute project path>`
> **Git:** <remote url or "no remote"> (<branch>)
> **Last scanned:** <date> via /arch_scan

```

**Change detection:** if `projects/<slug>.md` already exists, compare its current `## Architecture` body to the new `ARCH_MD` (ignore the header's date/git lines). 
- If the architecture body is unchanged, do not rewrite the file — just refresh the `Last scanned` date and report "no architecture change".
- If it changed (or the file is new), write the full file and report it as **added** or **updated**, and summarize what changed.

**Index:** maintain `/home/svankina/src/architect/projects/README.md` as a table. Create it if missing:

```markdown
# Project Architecture Index

Per-project architecture docs generated by `/arch_scan`. The architect repo is the source of truth; each project also carries a concise copy in its own `AGENTS.md`.

| Project | Source path | Last scanned |
|---------|-------------|--------------|
```

Add a row for the slug if absent, or update its `Last scanned` cell. Keep rows sorted by project name.

## 5. Report

Summarize concisely:
- Project scanned (slug + path).
- AGENTS.md: created / section updated / unchanged.
- Architect repo: `projects/<slug>.md` added / updated / unchanged, plus the index row.
- For an update, a one-line note on what shifted in the architecture.

Do **not** commit or push in either repo unless the user asks — just leave the changes staged in the working tree and tell them which files changed.
